# AWiki 开源文档风格与信息架构规范

[English](documentation-style-guide.md) | [简体中文](documentation-style-guide.zh-CN.md)

## 1. README 的职责

README 是项目首页，不是完整设计文档、API Reference 或发布 Runbook。

它必须按以下顺序回答问题：

1. **What**：这是什么？
2. **Who**：谁应该使用？
3. **Why**：解决什么问题，为什么不同？
4. **Status**：成熟度和关键边界是什么？
5. **Try**：如何在 5 分钟内获得第一次成功？
6. **Next**：详细文档、贡献与支持在哪里？

## 2. 标准 README 结构

```text
项目名与一句话价值主张
状态 / 许可证 / 核心技术 badges
截图或短演示
适用对象与核心价值
快速开始 / 第一次成功
主要能力
限制与兼容性
生态位置 / 架构概览
安全摘要
文档入口
贡献 / 支持 / 许可证
```

实现细节、全量配置表、测试矩阵、签名排障、完整命令树和 API 方法表应下沉到 `docs/`。

## 3. 文案要求

### 3.1 先写用户结果，再写实现

推荐：

> 在自己的域名运行一个 AWiki 兼容社区，提供身份、消息、附件与 ANP 互通。

不推荐：

> 一个基于 FastAPI、SQLite、JCS、Ed25519 与 JSON-RPC 的单进程服务。

技术栈可以在下一段或开发文档中说明。

### 3.2 每句话只承担一个主要信息

避免在首段连续列出十几个模块、协议名和内部边界。功能列表最多优先呈现 5–7 个用户可感知能力。

### 3.3 状态必须可验证

使用以下状态词：

| 状态 | 含义 |
| --- | --- |
| `MVP` | 用于验证核心闭环，能力和运维边界明显，不等于生产就绪 |
| `Developer Preview` | 面向开发者和早期测试者，接口或体验可能变化 |
| `Beta` | 主流程可用，但仍需兼容性与可靠性验证 |
| `Stable` | 有明确版本策略、升级契约、支持范围和持续验证 |
| `Experimental` | 不应作为默认生产路径 |
| `Unsupported` | 当前明确不可用，不用“部分支持”弱化 |

版本号本身不等于成熟度。

## 4. 命令示例规范

- 所有命令必须来自当前分支中的真实入口；
- 公共 README 中禁止出现未替换模板变量；
- 首个示例应尽量是可复制执行的最短路径；
- 有副作用的 CLI 操作优先展示 `--dry-run`；
- 命令后说明成功标准，而不是只展示命令；
- 使用占位值时采用 `<recipient-handle>`、`example.com` 等明显安全值；
- 不展示真实手机号、邮箱、DID、token、私钥、Team ID、服务器路径或测试账号。

## 5. 链接和文件命名

- 默认英文 README：`README.md`；
- 简体中文 README：`README.zh-CN.md`；
- 文件名和目录名使用英文 ASCII；
- 语言切换统一为：`[English](README.md) | [简体中文](README.zh-CN.md)`；
- 同仓库优先使用相对链接；
- 跨仓库使用固定的组织级 canonical URL；
- 不链接临时分支、个人 fork 或会过期的 CI artifact。

## 6. 截图和演示

- GUI 项目首屏必须有产品截图或 20–40 秒 GIF；
- CLI 项目应有一段真实终端演示；
- 服务端项目可使用终端演示、架构图和互通时序图；
- 截图必须有描述性 alt text；
- 截图中不得出现真实身份、token、手机号、邮箱、绝对路径或内部域名；
- 深色/浅色图片需要明确背景和对比度；
- 资源统一放在 `docs/assets/readme/`。

## 7. 兼容性写法

不要只写“支持 AWiki 客户端”。至少写清：

- 客户端或服务端版本；
- 已验证能力；
- 未验证能力；
- E2EE 状态；
- 平台状态；
- 验证日期；
- 是否需要 allowlist、特定域名或测试环境。

## 8. 安全文档要求

README 首屏只展示会影响采用决策的安全边界，例如：

- 是否支持 E2EE；
- 是否允许公网部署；
- 是否为单节点；
- 是否包含生产身份提供方；
- 哪些开发开关禁止用于公网。

详细密钥生命周期、SecretVault、签名、Keychain、权限和漏洞报告流程放在 `SECURITY.md` 与专项设计文档。

## 9. 维护流程

每次发布前：

1. 更新状态和版本兼容矩阵；
2. 在干净环境执行 README 中的所有命令；
3. 检查所有相对链接；
4. 检查截图与当前 UI 是否一致；
5. 检查英文、中文 README 的事实是否一致；
6. 搜索模板变量、TODO 和过期域名；
7. 让一位不了解项目的人完成 10 秒、60 秒和 5 分钟测试。
